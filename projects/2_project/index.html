Accessible Spanish Texts with LanguageTool

Enabling *Lectura F√°cil* (Easy-to-Read Spanish) through a custom authoring assistant

üß≠ Overview

Context.
Accessible text design for readers with cognitive or learning disabilities is essential‚Äîbut production of Easy-to-Read (E2R) or Spanish Lectura F√°cil (LF) content remains labor-intensive. Manual adaptation of texts involves rewriting for simpler vocabulary, shorter sentences, layout, and clarity. :contentReference[oaicite:1]{index=1}
This study investigated how a tool built on LanguageTool (an open-source writing assistant) could support adaptation of Spanish texts into LF by detecting guideline violations, offering simplifications and definitions. :contentReference[oaicite:3]{index=3}

My Role.
Lead researcher: defined the functional requirements of the tool (named LFWriteAssist), aligned LF guideline rules with LanguageTool custom rules, implemented vocabulary simplification modules, and documented limitations and user-flow design.

Timeline.
Approximately 3 months (tool prototype stage, usability testing and development of more rules still pending)

Team & Collaboration.
Partnered with linguistic accessibility experts at Hochschule Darmstadt (Germany) and HiTZ Center (Basque Country) :contentReference[oaicite:4]{index=4}

üéØ Research Question

How can LanguageTool be adapted into a Computer‚ÄêAssisted Translation (CAT) tool to support Spanish Lectura F√°cil (LF) content creation, and what are the benefits and limitations of such an approach?

üîç Methods

Approach.

  • Analyzed the UNE 153101:2018 EX LF standard for Spanish to identify vocabulary, syntax, layout, and style rules. :contentReference[oaicite:5]{index=5}
  • Created a set of custom rules in LanguageTool (via XML configuration) to flag LF-compliance issues: long words, figurative language, acronyms, complex sentences, etc. :contentReference[oaicite:6]{index=6}
  • Integrated external resources: Diccionario F√°cil (easy definitions for complex or polysemous terms) and EASIER corpus synonyms for readers with cognitive impairments. :contentReference[oaicite:7]{index=7}
  • Built an interface (LFWriteAssist) with three panels: input text, suggestions/definitions panel, and automatically corrected output. Rules applied colours (green = auto-changed, orange = flagged for review). :contentReference[oaicite:8]{index=8}
  • Conducted an internal evaluation: rule creation, tool deployment in prototype form; documented limitations (no user-testing yet). :contentReference[oaicite:9]{index=9}

Why this approach?

  • Combines linguistic guideline translation into automated rule-based checks (aligns with UX research focus on tooling and process).
  • Targets underserved user population (readers with cognitive or learning disabilities) which adds inclusive design emphasis.
  • Provides evidence of research-to-tool pipeline (useful for UX/Research portfolio to show process ‚Üí artifact).

üí° Key Findings & Insights

  • The prototype shows that a writing‚Äêassistant can substantially support the first pass of LF adaptation by flagging vocabulary, acronyms, and long phrases. :contentReference[oaicite:10]{index=10}
  • Some rules are straightforward to encode (e.g., avoid figurative speech, expand abbreviations), but others (e.g., ‚Äúavoid unnecessary words‚Äù, layout/style guidance) are difficult to formalize in rule-based systems. :contentReference[oaicite:11]{index=11}
  • The tool covers Spanish vocabulary and syntax, but does not yet address visual design/layout or user-interaction aspects of LF texts ‚Äî an important gap for full accessibility. :contentReference[oaicite:12]{index=12}
  • User validation remains pending: the tool is aimed at LF content creators, not direct readers with disabilities at this stage ‚Äî meaning that usability and reader comprehension outcomes are yet to be evaluated. :contentReference[oaicite:13]{index=13}

UX Takeaway.
Building accessible content requires not just automated rule-checking, but human oversight and user-centered evaluation. Tooling can expedite adaptation, but process design, user involvement, and layout/format still matter.

üß© Design Recommendations & Impact

Recommendations.

  • Integrate the tool into a browser/online platform to reduce installation barriers for accessibility practitioners. :contentReference[oaicite:14]{index=14}
  • Conduct usability testing with LF translators and target reader groups to assess tool effectiveness, workflow impact, and comprehension outcomes. :contentReference[oaicite:15]{index=15}
  • Extend rule-based system with layout/format checks (line breaks, sentence length, typography) and user-interaction components (e.g., read-aloud, adjustable font/spacing).
  • Develop a ‚Äúreview mode‚Äù for content creators: automation suggestions followed by human validation ‚Äî promoting a human-in-the-loop workflow.

Impact.

  • Demonstrates capability to bring linguistics, accessibility guidelines, and UX tooling together.
  • Useful example for product teams building assistive writing tools, text simplification workflows, or inclusive content pipelines.
  • In a UX research portfolio, this project shows methodical thinking, tooling implementation, inclusive-design awareness, and readiness to work across disciplines.

ü™û Reflections / Learnings

  • Working with guidelines: Translating textual guidelines into actionable rules is challenging ‚Äî you learn to balance precision vs ambiguity.
  • Tooling vs users: Building for accessibility means thinking beyond vocabulary: layout, comprehension, interaction all matter ‚Äî and tool alone is not enough.
  • Iterative mindset: The tool is a prototype ‚Äî next phases involve user feedback and real-world integration. I gained experience designing for underserved users and bridging technical implementation with user-centric goals.
  • If repeated: I would embed an end‚Äêuser study early (readers with cognitive disabilities) to measure comprehension gain and incorporate accessibility metrics (time to read, error rate) rather than only rule-compliance.
